## How The Pattern Breaks (And How To Fix It)

### When You'll Be Tempted To Abandon It

**1. When Research Feels Urgent**
**Temptation:** "I need to understand this library completely first"
**Fix:** Front-load research into handoff doc, THEN execute merry-go-round
**Example:** Today's handoff had alphaTab research done. Execution was pure momentum.

**2. When Scope Feels Too Small**
**Temptation:** "Just one file rendering? That's not impressive enough."
**Fix:** Ship the small thing. Impressive is the compound result, not individual phase.
**Example:** Phase 1 was just one file. Phase 4 was complete platform.

**3. When Perfectionism Strikes**
**Temptation:** "Let me optimize this before moving to next phase"
**Fix:** Working > perfect. Optimization is a separate phase.
**Example:** Phase 1 had large bundle. We shipped anyway. Optimize later if needed.

**4. When You Hit A Blocker**
**Temptation:** "Power through this 2-hour debugging session"
**Fix:** Document blocker, commit what works, pause. Merry-go-round thrives on wins, not grinding.
**Example:** Font loading failed. We fixed it in 5 minutes. If it took 30 min, we'd have documented and moved on.

**5. When Success Tempts Expansion**
**Temptation:** "This worked! Let me add 5 more features while I'm here!"
**Fix:** Ship what works. Note extras for future phases. Maintain focus.
**Example:** We didn't add file upload, zoom controls, or practice mode. We finished core first.
