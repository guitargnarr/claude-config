The guitar platform demonstrates what's possible when a solo developer collaborates with AI at scale. That's the truth. Not "I built this all myself" (false). Not "AI built this for me" (also false). But "I orchestrated this, AI executed it, together we built something neither could alone."

## On Substance

**Substance over flash.**

I'd rather build one thing that actually works than ten demos that look good in screenshots.

The guitar platform isn't about impressive UI or viral tweets. It's about:
- Does the search actually work? (Yes, debounced multi-field matching)
- Do the favorites persist? (Yes, localStorage with error handling)
- Does the progress track accurately? (Yes, tested across sessions)
- Does the architecture scale? (Yes, 8 modular components from 543-line monolith)

**Features should work reliably, not just look good.**

This value fights against my own impatience. The temptation to ship half-working code just to say "it's done." The urge to skip tests because they're tedious. The shortcut of sloppy architecture because "it works for now."

But shortcuts save time today and cost time every day after. Quality compounds.

Substance means:
- Research thoroughly before building
- Plan architecture, not just implementation
- Test everything—if it's not tested, it doesn't work
- Deploy confidently when quality is verified

No smoke and mirrors. No pretending. Show what's actually possible.

**Depth over appearance. Always.**

## On Building

**Modular craftsmanship.**

Components should:
- Have single, clear responsibility
- Compose cleanly with other components
- Work independently (testable in isolation)
- Use proper prop contracts (PropTypes, TypeScript)
- Handle their own state or receive it cleanly

This isn't just best practice—it's **values in action.**

Modular architecture forces:
- Clear thinking (tight coupling breaks parallel work)
- Better maintainability (each component understandable alone)
- Easier testing (isolated verification)
- Reduced conflicts (543-line file → 8 focused components = 25% → <5% conflict rate)

**The methodology is an expression of values.**

Research → Plan → Build → Test → Deploy isn't just process. It's:
- **Research** = Truth over comfort (understand before building)
- **Plan** = Substance over flash (architecture before implementation)
- **Build** = Modular craftsmanship (clean components, proper patterns)
- **Test** = Verify everything (no assumptions)
- **Deploy** = Confidence from quality (ship when it works)

Parallel development works **because** of these values. You can't parallelize sloppy architecture. You can't orchestrate 4 instances on poorly-specified work. The methodology requires the values.

**This is how I build: methodical, tested, scalable.**
